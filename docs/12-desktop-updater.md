# Desktop auto-updater

Hindsight Desktop ships with `tauri-plugin-updater`. On startup the app polls
GitHub Releases for a signed `latest.json`, and surfaces an in-app dialog if
a newer version is available.

## How it works

```
desktop app  ──HTTPS GET──▶  github.com/<repo>/releases/latest/download/latest.json
                                │
                                ▼
                       latest.json (+ per-platform .sig)
                                │
                          verify with embedded
                          public key (pubkey in
                          tauri.conf.json)
                                │
                                ▼
                       download installer ──▶  run passive NSIS
                                                installer ──▶  relaunch
```

- **Endpoint**: `https://github.com/m1abdullahh/screenshot-monitor-tool/releases/latest/download/latest.json` (configured in [tauri.conf.json](../apps/desktop/src-tauri/tauri.conf.json)). GitHub's `/releases/latest/...` path resolves to the most recent **published, non-draft, non-prerelease** release.
- **Signing**: every update bundle is signed with a minisign-style keypair generated by the Tauri CLI. The public half is embedded in [tauri.conf.json](../apps/desktop/src-tauri/tauri.conf.json); the private half lives only in GitHub Actions secrets and on the maintainer's machine.
- **Frontend UX**: [use-updater.ts](../apps/desktop/src/lib/use-updater.ts) drives state; [UpdaterDialog.tsx](../apps/desktop/src/components/UpdaterDialog.tsx) renders the prompt, progress bar, and "Restart now" button. Polls 30 s after boot, then every 6 hours.
- **Windows install**: NSIS in `passive` mode — small progress window, no admin prompt because we install per-user (`installMode: currentUser`). The installer kills and relaunches the app itself; the JS `relaunch()` path is the macOS/Linux fallback.

## One-time setup

### 1. Generate the signing keypair (do this once, on a trusted machine)

```bash
pnpm --filter @hindsight/desktop tauri signer generate -w "$HOME/.tauri/hindsight.key"
```

You'll be prompted for an optional password. **Use one** — without it, anyone who exfiltrates the key file can ship malicious updates to every installed copy.

This produces two files:

- `~/.tauri/hindsight.key` — **private key**. Never commit, never paste into chat, never email.
- `~/.tauri/hindsight.key.pub` — **public key**. Safe to commit.

### 2. Embed the public key in `tauri.conf.json`

Open `~/.tauri/hindsight.key.pub`, copy its contents, and replace the placeholder in [apps/desktop/src-tauri/tauri.conf.json](../apps/desktop/src-tauri/tauri.conf.json):

```diff
   "plugins": {
     "updater": {
-      "pubkey": "REPLACE_WITH_BASE64_PUBLIC_KEY_FROM_TAURI_SIGNER_GENERATE",
+      "pubkey": "dW50cnVzdGVkIGNvbW1lbnQ6IG1pbmlzaWduIHB1YmxpYyBrZXk...",
       ...
```

Commit and push. **The public key cannot be a file path** — it must be the literal key contents.

### 3. Upload the private key to GitHub Actions

In `https://github.com/m1abdullahh/screenshot-monitor-tool/settings/secrets/actions`, add two repo secrets:

| Secret name                          | Value                                                                  |
| ------------------------------------ | ---------------------------------------------------------------------- |
| `TAURI_SIGNING_PRIVATE_KEY`          | Full contents of `~/.tauri/hindsight.key` (multi-line, paste as-is)    |
| `TAURI_SIGNING_PRIVATE_KEY_PASSWORD` | The password you set in step 1 (or empty string if you didn't set one) |

The workflow at [.github/workflows/desktop-release.yml](../.github/workflows/desktop-release.yml) consumes these as env vars on the `tauri-action` step. They are never written to disk by CI and never reach the frontend bundle.

### 4. Verify the first signed release

```bash
git tag desktop-v0.1.1
git push origin desktop-v0.1.1
```

The workflow will build all three platforms, sign each installer, generate `latest.json`, and attach everything to a **draft** release. Inspect the draft to confirm:

- `latest.json` is present in the release assets
- Each platform has both the installer and a `.sig` file
- Opening `latest.json` shows real signature blobs (not empty strings)

Promote the draft to **published** (uncheck "Set as a pre-release", click Publish). At that point the `/releases/latest/...` URL resolves and existing installs will start picking up the update on their next poll.

## Rolling out updates

1. Bump `version` in [apps/desktop/src-tauri/tauri.conf.json](../apps/desktop/src-tauri/tauri.conf.json) and [apps/desktop/package.json](../apps/desktop/package.json).
2. Tag and push: `git tag desktop-v0.1.2 && git push origin desktop-v0.1.2`.
3. Wait for the workflow to finish (~15 min for all three platforms).
4. Edit the resulting draft release: paste in release notes (they become `notes` in `latest.json` and show up in the in-app dialog), then click **Publish release**.

Installs poll every 6 h after boot (and 30 s after the first launch), so the rollout reaches everyone within a day. If a user has the app open continuously, they'll see the prompt on the next poll.

## Key rotation

If the private key is ever exposed:

1. Generate a new keypair (`pnpm tauri signer generate`).
2. Ship a **regular code update** that swaps the `pubkey` in `tauri.conf.json` to the new public key, signed with the **old** private key. Every install picks this up.
3. Wait for adoption (check `/auth/me` heartbeats or whatever telemetry).
4. Rotate the GitHub secrets (`TAURI_SIGNING_PRIVATE_KEY`, `TAURI_SIGNING_PRIVATE_KEY_PASSWORD`) to the new key.
5. From the next release onward, only the new key can sign updates.

If users skip the rotation update (e.g. uninstalled, reinstalled an old version), they're stuck on whatever they have — there is no out-of-band way to re-bless a new public key.

## Local testing without GitHub

`tauri-plugin-updater` accepts arbitrary endpoints. To dry-run against a local file server:

```bash
# Build with createUpdaterArtifacts on (already configured)
TAURI_SIGNING_PRIVATE_KEY=$(cat ~/.tauri/hindsight.key) \
  TAURI_SIGNING_PRIVATE_KEY_PASSWORD=yourpw \
  pnpm --filter @hindsight/desktop tauri build
```

Then hand-author a `latest.json` pointing at the resulting `.msi.sig` / installer URLs hosted via `python -m http.server`, and temporarily swap the endpoint in `tauri.conf.json`. Don't ship that swap.

## Things to be careful about

- **Draft vs published**: the `/releases/latest/...` URL skips drafts and pre-releases. A signed `latest.json` on a draft release is invisible to clients until you click Publish.
- **Per-user install scope**: the NSIS installer is `currentUser`, so updates install for the same user that ran the app. If somebody installed for "All Users" by hand, updates will silently do nothing.
- **CSP**: the renderer's `connect-src` includes `https://github.com` and `https://objects.githubusercontent.com` so the updater's progress events (which run inside the webview) can hit the redirect target. Don't tighten these without checking the updater still works.
- **Version comparison**: by default Tauri refuses to downgrade. If you ever need to push users to an older version (rollback), use the Rust `updater_builder().version_comparator(...)` escape hatch.
